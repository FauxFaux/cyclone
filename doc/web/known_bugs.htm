Cyclone Implementation Bugs

Version 1.0

  • Potential name clashes

  • Variables in exception handlers may have the wrong value

  • ints and pointers must be 32 bits.  (This restriction is not pervasive in the compiler, so we should be able to support a 64-bit architecture with a bit of work.)

  • casts with respect to alignment on struct fields

  • We generally do not detect null pointer exceptions, so you'll seg fault.

  • gcc must be in Cyclone's path and called gcc b/c of pre-processor call (should allow redirection somehow)

  • type declarations can only appear at top-level

  • type qualifiers (eg. const) are not yet checked for correctness in the Cyclone type-checker.

  • Should Null_Exception by Cyc_Null_Exception?

  • Switch clauses should be checked in the other order (a pain b/c of fallthru)

  • Array-bounds errors should throw Array_Bounds, not Null_Exception.

  • Subtraction of t? types is translated incorrectly.

  • Link-checker (or somebody) should check the type of Cyc_main.

  • We don't actually prevent a tunion with an enormouse number of non-value-carrying variants.

  • Pattern-matching against a pointer to tunion Foo.Bar where tunion Foo.Bar carries variants is busted.

  • Non-uniqueness of existential tyvars allows capture?

  • extern "C" clash because only translation (and not type-checker) strips namespaces

  • structs with array members -- initialization gets translated incorrectly

  • Ways to crash: allocate too much (eg. new { for i < -1 : 0 }), stack overflow, divide-by-zero
